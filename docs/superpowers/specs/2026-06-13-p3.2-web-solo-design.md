# P3.2 — Solo client-local web (design)

> Design validé le 2026-06-13. Compagnon du `contrat-pazaak.md` (§5-6, §11),
> `RULES.md`, `ROADMAP.md` (P3). Critère de sortie P3 : **partie solo complète jouable
> au navigateur contre l'IA**. La logique de règles reste **exclusivement** dans
> `packages/engine` ; le web ne fait que piloter et afficher.

---

## 1. Objectif & périmètre

Câbler `apps/web` (TanStack Start, aujourd'hui un placeholder nu) pour offrir une partie
solo complète contre l'IA, en client-local `boardgame.io` (aucun réseau, aucun crédit,
aucune server function — celles-ci relèvent de P4).

**Dans le périmètre**
- Fondation app shell TanStack Start **réutilisable** (root layout, routing) pour accueillir
  lobby/auth/leaderboard en P4/P6.
- Écran d'accueil avec sélecteur de **difficulté** (mappée sur `AiParams.standThreshold`).
- **Deck-builder** : le joueur humain compose ses 10 cartes parmi le catalogue standard.
- Écran de **plateau** (board) : pioche/score/main/moves, plateaux des deux joueurs.
- Écran de **fin** de set/match (vainqueur, sets gagnés, rejouer).
- Pilotage de l'IA sur le siège `'1'` (Approche A, cf. §4).
- `data-testid` sur tout élément interactif (prépare l'e2e P7, scénarios 2 et 4).

**Hors périmètre**
- Server functions, SQLite, auth, crédits, lobby, ranked (P4+).
- Gold cards (P8). L'IA et le deck-builder ne manipulent que des cartes standard.
- Animations, sons, écran de règles (P8 polish).
- Sélecteur de difficulté avancé, decks pré-définis, quantités/doublons dans le deck-builder
  (raffinements P8).

## 2. Architecture & routing

Routing file-based TanStack Start (`@tanstack/react-router` 1.170.15,
`@tanstack/react-start` 1.168.25, épinglés) :

```
apps/web/src/
├── router.tsx / config Start      # plomberie TanStack Start + Vite plugin
├── routes/
│   ├── __root.tsx                 # app shell : header, layout, <Outlet/> (rails P4/P6)
│   ├── index.tsx                  # accueil : "Jouer en solo" + sélecteur de difficulté
│   └── solo.tsx                   # route du match solo (difficulté en search param typé)
├── solo/
│   ├── clients.ts                 # factory des 2 clients Local() (humain '0', IA '1')
│   ├── use-solo-game.ts           # souscription client humain → vue strippée + moves + écran
│   └── use-ai-driver.ts           # Approche A : pilotage du client IA
├── components/
│   ├── MainMenu.tsx
│   ├── DeckBuilder.tsx
│   ├── Board.tsx
│   ├── GameOver.tsx
│   ├── CardView.tsx               # side card / carte posée
│   ├── PlayerPanel.tsx            # plateau + score + sets + statut d'un joueur
│   └── HandView.tsx               # main du joueur humain
└── store.ts                       # Zustand : état UI pur (buffer deck-builder)
```

Le `__root` porte l'app shell réutilisable (header, conteneur, providers). La difficulté
est choisie sur `/` et passée à `/solo` via search param typé (`?difficulty=normal`),
mappé en `standThreshold` : `easy → 19`, `normal → 18`, `hard → 17` (défaut `normal`).

## 3. Cycle de vie & écrans

À l'entrée de `/solo`, on crée **deux** clients `boardgame.io` `Local()` partageant le
`matchID` : humain (`playerID: '0'`) et IA (`playerID: '1'`), `numPlayers: 2`. Les clients
vivent toute la durée du match. **L'écran affiché est dérivé de l'état**, pas une route :

| Condition (vue du client humain) | Écran |
|---|---|
| phase `pickSideDeck`, humain pas encore pické (`players['0'].sideDeck === null`) | `DeckBuilder` |
| phase `play`, `matchWinner === null` | `Board` |
| `matchWinner !== null` | `GameOver` |

**Deck-builder** : le catalogue expose 18 cartes distinctes (6 valeurs × 3 signes
`+`/`-`/`±`, via `standardSideCardCatalogue()`). Le joueur en sélectionne **10 distinctes**
(compteur `n/10`, bouton « Valider » actif à 10). À la validation :
`humanClient.moves.pickSideDeck(cards)`. Le buffer de sélection vit dans Zustand (état UI),
confirmé vers l'engine au clic.

**Board** : rendu depuis la vue strippée du client humain. Affiche les deux `PlayerPanel`
(plateau, score, sets gagnés, statut standing/busted), la `HandView` du joueur humain, et
les actions du tour (`playCard` avec déclaration de signe pour les ±, `endTurn`, `stand`).
L'adversaire est rendu via `OpponentView` (`hand: { count }`, plateau/score/sets publics) —
sa main n'est jamais dans le DOM.

**GameOver** : vainqueur (`matchWinner`), score en sets, bouton « Rejouer » (recrée des
clients neufs / réinitialise la route).

## 4. Pilotage de l'IA (Approche A — deux clients + driver manuel)

L'IA est une fonction **pure** d'`engine` (`chooseMove(self, params)`,
`chooseSideDeck()`), totalement découplée du runtime. Le web la pilote via un second
client headless.

`use-ai-driver.ts` s'abonne au client IA ; à chaque notification :

1. **Phase `pickSideDeck`** : si `'1'` est actif au stage `pick` et `sideDeck === null`
   → `aiClient.moves.pickSideDeck(chooseSideDeck())`.
2. **Phase `play`** : si `ctx.currentPlayer === '1'` et `'1'` actif (ni `standing`, ni
   `busted`) → lire son `players['1']` (vue complète de son propre siège), calculer
   `const m = chooseMove(self, { standThreshold })`, puis dispatch **générique**
   `aiClient.moves[m.move](...m.args)`.

`AiMove` étant uniforme (`args` toujours présent, `[]` pour stand/endTurn), le dispatch
n'a aucun cas particulier.

**Garde-fous** : un flag de ré-entrance (`isActingRef`) + revérification de l'état avant
dispatch évitent les doubles coups sur notifications rapprochées. Le moteur autorisant un
joueur courant « actif > 20 » (bust-recovery, cf. QUIRKS), le driver fournit **toujours**
un coup (rescousse via side deck, sinon `endTurn` qui finalise le bust) — il n'y a pas
d'auto-fin de tour sur dépassement.

**Rejet de B (bots natifs `Local({ bots })`)** : l'API `Bot` boardgame.io vise
l'énumération/MCTS, pas notre heuristique pure ; la signature contractuelle est
`chooseMove(PlayerState)`. L'ordre de tours custom, la phase pick et la rescousse rendraient
le wrapper artificiel, pour un couplage framework accru et aucun gain de qualité.
**Rejet de C (piloter l'IA dans la définition du jeu)** : violerait la pureté d'`engine`
et le contrat (« l'IA est pure, le web la pilote »).

## 5. Découpage de l'état

- **Client `boardgame.io`** = source de vérité du jeu (G, moves, phases). Aucune règle,
  aucun scoring, aucune validation de move dupliqués côté web (règle absolue CLAUDE.md).
- **Hook React** (`use-solo-game`, `use-ai-driver`) = pont de souscription
  (`client.subscribe` → state React).
- **Zustand** (`store.ts`) = état UI pur uniquement (buffer de sélection du deck-builder
  avant confirmation, et éventuellement la difficulté si non portée par la route). Le reste
  vient de l'engine.

Chaque client ne lit que **sa** vue strippée (`playerView`) : le secret state (main/side
deck de l'IA) est garanti par l'engine, jamais reconstruit côté UI.

## 6. Gestion d'erreur

Périmètre solo, sans réseau ni persistance — pas de gestion réseau/DB (P4+).

- **Coups illégaux** : l'UI **désactive en amont** les actions impossibles selon l'état
  (`playCard` grisé si `playedHandCardThisTurn`, déclaration de signe obligatoire avant
  dispatch d'une ±, actions masquées si standing/busted), plutôt que d'attendre un
  `INVALID_MOVE`. Un `INVALID_MOVE` reçu = bug de pilotage à corriger, pas un cas
  utilisateur.
- **Driver IA** : `chooseMove` est total (renvoie toujours un `AiMove`) — pas de chemin
  « l'IA bloquée ». Un garde défensif logge si un dispatch IA était rejeté (ne devrait
  jamais arriver).

## 7. Stratégie de test

- **Engine** : déjà couvert (50 tests + fast-check 1000 runs, IA pure testée). On n'y
  touche pas. Le seuil de couverture 90% du job CI `quality` ne porte que sur `engine`.
- **Test d'intégration « deux sièges pilotés par l'IA »** (suspens noté au HANDOFF) :
  test headless dans `apps/web` qui crée les 2 clients `Local()` seedés, branche le
  pilotage des deux côtés (ou pilote `'0'` par script + IA sur `'1'`), et joue jusqu'à
  `matchWinner`. Valide la boucle de pilotage de bout en bout **sans navigateur**.
- **Pas de tests de composants** : décision ROADMAP — l'UI est couverte par les e2e P7
  (scénario 2 « solo » et 4 « secret state »). Les `data-testid` sont posés dès P3.2.
- **Définition de « terminé »** : `command pnpm exec biome check .`, `pnpm -r typecheck`,
  `pnpm -r test` verts à la racine + CI GitHub Actions verte (jobs `quality` + `security`).

obsolètes).

mises à jour (INDEX, HANDOFF, et QUIRKS/ENVIRONMENT/CONVENTIONS si découvertes).
